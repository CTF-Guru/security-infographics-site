PASSKEYS

01
What a passkey is — a FIDO2 / WebAuthn public-key credential that replaces the password

A passkey is a public/private key pair — not a secret you type. The private key lives in your device's authenticator; the server keeps only the matching public key.

02
Registration — create a passkey
device makes a key pair→ public key → server

The pair is bound to the site's origin (the RP ID). The public key registers with the server; the private key never leaves the device.

03
User verification — stays local
👆
Biometric or PIN unlocks the key — on the device, in the authenticator. This is user verification.
🚫
Nothing biometric is sent — your fingerprint or face never leaves the device or reaches the server.

The gesture proves it's you to your own device — the server only ever sees a signature.

04
Sign-in — challenge → unlock → sign → verify

Only a signature over a one-time challenge crosses the wire — the server verifies it with the public key it already holds.

05
Synced vs device-bound

☁️ Synced

Roams across your devices via a provider keychain.Apple · Google · 1Password

🔐 Device-bound

A hardware key — the private key never leaves it.YubiKey · security key

Synced = easy roaming & recovery · device-bound = highest assurance.

06
Cross-device sign-in — hybrid

Sign in on a laptop using the passkey on your phone:

scan the QR→ Bluetooth proximity→ phone signs

The proximity check blocks a remote relay — the phone must be nearby.

07
Why passkeys beat passwords — nothing to steal
🎣
Nothing to phish — the key is origin-bound; a look-alike site can't get a valid signature.
♻️
Nothing to reuse — a unique key pair per site, so one leak never unlocks another.
🗄️
Nothing to breach — the server holds only public keys, useless to an attacker who dumps them.
⌨️
Nothing to type — just a tap and a biometric; no secret to enter, mistype, or forget.
08
The moving parts
🔑
Authenticator — holds the private key & signs; platform (phone / laptop) or roaming (a hardware key).
🌐
WebAuthn API — the browser API the site calls to create & use credentials.
🏛️
Relying Party — the site; issues the challenge & verifies the signature.

FIDO2 = WebAuthn (site ↔ browser) + CTAP (browser ↔ authenticator).

09
Passwordless
🚀
Replace the password — phishing-resistant on its own, so it can log you in with no password at all (true passwordless).
🧩
Or add it as a factor — a passkey can also serve as a strong second factor alongside an existing login.

User verification makes one passkey both possession and inherence in a single gesture.

10
On the wire — the WebAuthn assertion the browser returns
// navigator.credentials.get() — device signs the server's random challenge; nothing secret is sent
{ "id": "<credential-id>", "type": "public-key", "response": { "clientDataJSON": "{ challenge, origin, type }",
  "authenticatorData": "<rpIdHash · flags(UP,UV) · signCount>", "signature": "<over the two>" } }
// server verifies the signature with the stored public key — the origin binding defeats phishing
11
Watch out
🚪
Recovery is the hard problem — account reset is where attackers push. Design & harden it up front.
🧳
Portability & migration — moving passkeys across ecosystems is still uneven; plan for it.
🪜
Keep a fallback — a lost device shouldn't lock you out; register a backup passkey or key.
🏢
Attestation & policy — enterprises may require device-bound keys and attestation to trust a credential.
☁️
Synced trust rests on the provider — a synced passkey is only as safe as the keychain account it roams through.
12
The takeaway
A passkey is a cryptographic key bound to the real site and unlocked by you — no password to phish, reuse, or breach.
Key pair per site→ Unlock locally→ Sign the challenge→ Verify with public key
