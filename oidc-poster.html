OPENID CONNECT

The identity layer on OAuth 2.0 — proving who signed in with a verifiable ID token (the engine behind “Sign in with Google”)
CTF Farooq Ali
· security explainer series
v2026.06
01
What OIDC adds — identity, on top of OAuth

Ask for the openid scope and OAuth hands back an ID token — signed proof of the user's identity.

02
The three roles — in OIDC's words

👤 End User

The person signing in — the identity being proven.

📱 Relying Party (RP)

Your app — it relies on the provider to authenticate the user.

🔐 OpenID Provider (OP)

Google, Microsoft, Okta… authenticates & issues the ID token.

“Sign in with Google” — Google is the OP, your app is the RP.

03
The sign-in flow — Authorization Code + PKCE, with scope=openid

Same flow as OAuth — the openid scope is what adds the ID token in step 5.

04
The ID token — a signed JWT
header. payload. signature

Claims: iss (issuer) · sub (stable user id) · aud (which app) · exp (expiry) · nonce.

Proof of identity — verify it, don't just read it.

05
ID token ≠ Access token
🪪
ID token — for the app. Proves who logged in. Its aud is your client.
🎟️
Access token — for the API. Grants access. Sent as Bearer.

Never send the ID token to an API, nor use an access token to identify the user.

06
Scopes → claims

openid is required — it's what makes it OIDC. Add more for standard claims:

openidprofileemailaddressphone

Need more detail? Call the UserInfo endpoint with the access token.

07
Validate the ID token

Decoding ≠ trusting. Check every time:

signature (JWKS)· iss· aud· exp· nonce

A token you didn't verify is just a string an attacker can forge.

08
On the wire — the /token response now carries an id_token
// /token response — access_token as before, plus the id_token
{ "access_token": "eyJhbGc…", "id_token": "eyJhbGc…<JWT>", "token_type": "Bearer", "expires_in": 3600 }
// the id_token payload, decoded — claims about the user
{ "iss": "https://accounts.google.com", "sub": "10769150350006150", "aud": "<your-client-id>",
  "exp": 1699999999, "email": "ada@example.com", "nonce": "n-0S6_…" }
09
Discovery

The provider publishes everything at one well-known URL:

GET /.well-known/openid-configuration

→ the endpoints (authorize · token · userinfo) and jwks_uri — the keys you verify with.

10
Watch out
🎭
Access token as login — it proves access, not identity. Use the ID token.
✍️
Unverified signature — always check it against the provider's JWKS keys.
🎯
Ignoring aud — a token minted for another app isn't yours to trust.
🆔
email as the key — emails change & get reused. Key users on sub + iss.
11
The takeaway
OIDC is OAuth + identity — one openid scope turns delegated access into a verifiable proof of who signed in.
openid scope→ ID token→ Validate→ Know the user
