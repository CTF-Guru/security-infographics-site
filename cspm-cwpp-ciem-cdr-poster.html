CLOUD SECURITY PILLARS

CSPM · CWPP · CIEM · CDR — four lenses on cloud risk: is it configured right, are the workloads protected, who can do what, and is something attacking now (usually bundled into one CNAPP)
CTF Farooq Ali
· security explainer series
v2026.06
01
Four questions about cloud security — one pillar answers each

Three build-and-config pillars — CSPM, CWPP, CIEM — plus one runtime detection pillar, CDR. Together they cover posture → protection → identity → detection.

02
CSPM — posture / config

Cloud Security Posture Management — audits the cloud account against best practice & policy.

🪣
Exposed storage — public buckets, open snapshots, world-readable data.
🔓
Weak network rules — 0.0.0.0/0 security groups, no encryption at rest.
📐
Policy drift — config wanders from the baseline across accounts.

Finds misconfigurations — before an attacker does.

03
CWPP — workload / runtime

Cloud Workload Protection Platform — protects the things that actually run.

🖥️
Covers — VMs, containers & serverless functions, wherever they live.
🐞
Vuln scanning — known CVEs in images, packages & the OS.
🛡️
Runtime protection — catches malicious behaviour as the workload runs.

Guards the workload — build-time scan + run-time defence.

04
CIEM — identity / entitlements

Cloud Infrastructure Entitlement Management — untangles who can touch what in cloud IAM.

👑
Over-privileged — identities granted far more than they ever use.
☣️
Toxic combos — permission pairs that together enable escalation.
🔗
Risky role chains — A assumes B assumes C → an unexpected path to power.

Right-sizes identity — trims unused & excess privilege.

05
CDR — detection / response

Cloud Detection & Response — the runtime watchtower for an attack in progress.

📜
Cloud logs — control-plane & API activity (CloudTrail-style events).
🪪
Identity signals — odd logins, sudden privilege use, key abuse.
🧭
Attack-path context — stitches events into one narrative, not alerts.

Detects active threats — and drives the response.

07
How they stack — three foundations under one runtime lens

Posture, protection & identity feed the signals; CDR turns them into detection.

08
Why together — one real breach chains all three
Misconfig — public bucket (CSPM)→ Vulnerable workload — unpatched (CWPP)→ Over-privileged role assumed (CIEM)→ CDR catches it in progress

One pillar alone sees only its slice — it misses the chain. That is exactly why a CNAPP bundles CSPM + CWPP + CIEM + CDR into one platform that follows the attack path end to end.

09
Watch out
🏷️
Overlapping claims — one vendor's "CNAPP" may be light on a pillar. Check coverage, not the label.
🔀
CSPM ≠ CWPP — config posture is not runtime protection. A clean config still runs vulnerable code.
🙈
CIEM is overlooked — cloud IAM is huge & permissive by default; entitlement sprawl hides in plain sight.
10
Which pillar? — name the question

Is it set up right?

CSPM — audit config & exposure.

Is the code safe?

CWPP — scan & guard workloads.

Who can do what?

CIEM — right-size entitlements.

And "is anything happening now?" → that's CDR, watching at runtime.

11
The takeaway
CSPM checks config, CWPP guards workloads, CIEM controls identity, CDR catches attacks — four lenses on cloud risk, usually one CNAPP.
